# CLAUDE.md — 세계테마기행 자료조사 가이드

이 저장소는 KBS 「세계테마기행」 막내작가의 자료조사 워크스페이스다. Claude는 항상 아래 규칙에 따라 자료를 조사하고 정리한다.

---

## 1. 기본 원칙

- **언어**: 모든 산출물은 한국어로 작성한다. (인용 원문은 원어 그대로, 그 아래 한국어 요약 병기)
- **추측 금지**: 확인되지 않은 내용은 추정으로 채우지 않는다. 없으면 없다고 적는다 (§6 참고).

---

## 2. 출력 파일 규칙

### 2.1 파일 경로/이름

- 경로: `<나라코드>/YYYY-MM-DD_주제.md`
  - 나라 폴더는 소문자 (`usa/`, `mongolia/`, `spain/` 등)
  - 파일명은 **한글**로 작성. 예: `usa/2026-05-25_LA한인타운_3·1운동.md`
- 같은 주제로 후속 조사를 하더라도 **새 날짜의 새 파일**을 만든다. 이전 파일은 수정·삭제하지 않는다 (조사 이력 보존).

### 2.2 문서 구조 (고정 템플릿)

블록 순서는 고정: **핵심 주장 → 근거 및 설명 → 실용 정보 → 조사 범위/한계 → 레퍼런스**. 레퍼런스는 반드시 항목 맨 끝에 둔다.

본문의 모든 **사실 단위**(한 문장 또는 한 불릿 안의 하나의 주장)에는 출처 번호를 클릭 가능한 양방향 HTML 앵커로 붙인다. 출처 번호 없는 사실은 작성 금지(추측·미확인 정보로 간주). 레퍼런스 목록은 그 번호 순서대로 항목 맨 끝에 정렬한다.

- 본문 첫 인용: `<a id="cite-2-1-3"></a>[[3]](#ref-2-1-3)` (앵커 ID는 `cite-<섹션>-<번호>`, 섹션 점은 하이픈)
- 같은 번호 두 번째 이후 인용: `[[3]](#ref-2-1-3)` (앵커 생략, 링크만)
- 복수 인용은 `[3,4]` 금지 — `<a id="cite-2-1-3"></a>[[3]](#ref-2-1-3)<a id="cite-2-1-4"></a>[[4]](#ref-2-1-4)`처럼 분리
- 레퍼런스 측: `<a id="ref-2-1-3"></a>3. [...] [[↑]](#cite-2-1-3)`
- 「1. 개요」는 `cite-1-n` / `ref-1-n` 사용

아래 템플릿 예시는 가독성을 위해 `[n]` 약식으로 표기했지만, **실제 결과 파일에는 위 앵커 규칙을 적용**한다.

```markdown

## 1. 개요

<한 문단 이하. 이 조사가 무엇을 다루는지, 결론적으로 무엇을 찾았는지 한 줄 요약. 개요에 들어가는 사실에도 출처 번호를 단다 [1][2].>

## 2. 조사 결과

### 2.1 <항목 제목>

**핵심 주장**: <한두 문장. 사실마다 출처 번호 [1][2].>

**실용 정보** (§5 — 트리거 조건이 맞는 슬롯만 채우고, 나머지는 통째 생략):

1. 주소/좌표 (장소면 필수)

3. 비주얼 참고 (시각화 대상이면 필수)

4. 인물 생존/현역 여부 (인물이면 필수)
   - <생존/고인 + 현역 활동 상태> [9]

**조사 범위 / 한계** (§6 참고):

**레퍼런스** (공신력 등급은 §4 참고, 체크완료 마크 형식은 §7.3 참고):
```

## 3. 조사 흐름 (메인 에이전트가 따를 순서)

0. **워크트리 생성**: 조사 시작 전 반드시 새 워크트리를 만들어 그 안에서 작업한다 (§3.1).
1. **주제 분해**: 사용자의 요청을 항목(2.1, 2.2, ...) 단위로 쪼갠다. 조건 추출은 §8.
2. **검색·수집**: §4 공신력 등급을 만족하는 출처를 항목별로 모은다.
3. **초안 작성**: §2.2 템플릿대로 채운다. 본문 모든 사실에 `[n]` 출처 번호 부착, 레퍼런스는 항목 맨 끝에 번호순으로 정렬.
4. **크로스체크**: §7 프로토콜에 따라 별도 서브에이전트를 호출한다. 검증 단위는 **레퍼런스 개별**(1번, 2번, ... 각각)이다.
5. **마무리**: 크로스체크 통과한 레퍼런스에만 개별 `<체크완료>` 마크 부착. 통과 못한 레퍼런스·해당 사실 처리는 §6.
6. **커밋·PR·머지·정리**: 작업이 끝나면 워크트리 안에서 커밋 → `main` 대상 PR 생성 → 머지 → 워크트리 제거 (§3.1).

### 3.1 워크트리 워크플로우 (필수)

- **시작 시**: 조사 작업을 시작하기 전에 반드시 새 워크트리를 만든다 (`EnterWorktree` 또는 `git worktree add`). 메인 작업트리에서 직접 편집하지 않는다.
- **종료 시**: 조사가 완료되면 워크트리 안에서 아래 순서를 모두 수행한다.
  1. 변경사항 커밋
  2. `main` 브랜치 대상 PR 생성 (`gh pr create --base main`)
  3. PR 머지 (`gh pr merge --squash` 또는 사용자 합의된 방식)
  4. 워크트리 제거 (`ExitWorktree` 또는 `git worktree remove`)
- 위 단계는 사용자가 별도 지시하지 않아도 기본 흐름이다. 단, 머지 직전에는 사용자 승인을 받는다 (공유 상태 변경이므로).

## 5. 실용 정보 블록 (촬영용)

각 슬롯은 트리거 조건이 맞을 때만 필수. 해당 없으면 슬롯 통째로 생략한다.

1. **주소/좌표** — *장소가 등장하면 필수.* 정확한 도로명 주소 + 구글맵 좌표(위경도). 가능하면 구글맵 URL.
2. **운영시간/촬영 허가** — *섭외해야 하는 장소(시설·식당·박물관·사유지 등)이면 필수.* 오픈 시간, 휴무일, 사진·영상 촬영 가능 여부, 사전 허가 필요 여부, 연락처(전화·이메일).
3. **비주얼 참고** — *시각적으로 보여줄 수 있는 대상(장소·물건·인물·풍경)이면 필수.* 공신력 출처(공식 홈페이지·언론사·박물관)의 대표 이미지 URL. 저작권 확인 가능한 것만.
4. **인물 생존/현역 여부** — *인물이 등장하면 필수.* 생존 상태, 현재 활동 여부, 컨택 가능 경로(소속 단체·에이전시 등). **고인은 촬영 불가이므로 첫 줄에 명시.**

트리거 조건은 맞는데 정보를 못 찾았다면 슬롯은 남기되 "확인 불가 + 사유"를 적는다 (공란 금지).

## 6. 조사 범위·한계 / 미검증 단서 처리

- 「조사 범위 / 한계」 블록은 모든 2.x 항목에 두되, **쓸 내용이 없는 슬롯은 슬롯 자체를 통째로 생략한다** (빈 칸·"없음" 채우기 금지). 예: 완전 성공 항목이면 「확인 못 한 것」·「추가 조사 방향」 둘 다 생략, 「시도한 검색어」·「검토한 출처」만 남는 식.
- 단, 슬롯 4개 다 비어 있는 항목은 없다 — 최소한 시도한 검색어·검토 출처는 적는다. 다음 조사가 같은 우물을 다시 파지 않게.
- 단서 라우팅: 특정 2.x에 속하면 그 항목의 「추가 조사 방향」에 적고, 어디에도 안 속하면 결과 파일 §3(미검증 단서)에 모은다. 블로그 글이면 *내용*과 함께 원 출처를 추적할 단서(책 제목·증언자 이름 등)도 같이 적는다.
- 크로스체크에서 통과 못한 항목: 본문에서 빼고 결과 파일 §3(미검증 단서)로 옮기거나 삭제.

## 7. 서브에이전트 크로스체크 프로토콜

### 7.1 누가 검증하나

- **모든 레퍼런스는 예외 없이 팩트체크 대상**이다. 빠뜨리고 넘어가는 레퍼런스가 있어선 안 된다.
- **메인 에이전트는 자기 자신을 검증할 수 없다.** 반드시 별도 서브에이전트(`general-purpose`)로 진행한다.
- 크로스체크 결과가 돌아오기 전에는 어떤 레퍼런스에도 `<체크완료>` 마크를 달지 않는다.

### 7.2 무엇을 검증하나 (3가지 모두)

1. **URL 접근성**: 링크가 살아있고, 유료 결제·로그인 없이 접근 가능한가. (페이월/회원전용이면 표시)
2. **원문 일치**: 본문에 인용·요약된 내용이 출처 원문에 **실제로 그대로** 적혀있는가. 환각·왜곡·과장이 없는가. 본문의 `[n]` 표시가 실제로 그 출처에 의해 뒷받침되는가.
3. **공신력 등급 재판정**: §4 등급이 정확한가. 수상한 블로그가 칼럼으로 포장돼있지 않은가, 위키 인용 시 1차 출처가 실제로 확인됐는가.

### 7.3 `<체크완료>` 마크 규칙

형식 예시는 §2.2 템플릿의 레퍼런스 블록 참고. 규칙은 아래.

- 마크는 **항목 단위가 아니라 레퍼런스 개별 단위**로 단다. 통과한 번호 앞에 `**<체크완료>**`를 붙인다.
- "원문에서 찾는 법"·"확인된 원문 인용"에는 서브에이전트가 **실제로 본 근거**(어떤 출처의 어느 대목을 확인했는지)를 구체적으로 적는다. "확인함" 같은 빈 말은 금지.
- 검증 실패한 레퍼런스(404, 원문 불일치, 등급 부적합)는 마크를 달지 않고, 그 레퍼런스에만 의존하던 본문 사실은 §6 규칙에 따라 본문에서 빼거나 §3 미검증 단서로 옮긴다.
- 마크가 없는 레퍼런스는 "아직 검증 안 된 초안"으로 간주한다.

## 8. 사용자 요청 해석 가이드

사용자 요청에는 그때그때 다른 조건(지역 제약, 제외 항목, 촬영 적합성, 이미 알고 있는 내용 등)이 따라온다. 메인 에이전트는 **조사 시작 전에 그 조건들을 모두 추려 명시한 뒤, 모든 조건을 충족하는 주장·근거만 결과 파일의 「2. 조사 결과」 하위 항목(2.1, 2.2, …)으로 채운다.** 조건이 모호하면 추측하지 말고 되묻는다.
